# MomentumPullbackEA v1.25 audit + v1.26.x safety patch (2026-07-10)

38-agent adversarially-verified audit of the live FTMO build (`mql5/MomentumPullbackEA.mq5`,
renamed from DerivScalperEA): 47 raw findings → 29 CONFIRMED, 2 refuted, 0 uncertain.
Entry math, lot sizing, and bracket placement passed clean; defects clustered in
state persistence, halt semantics, one MQL5 API misuse, and live-vs-validated parity.

## P1s (verified)

1. **ATR estimator parity** — MT5 `iATR` is a sliding **simple** mean of TR
   (terminal's own `Indicators/Examples/ATR.mq5` line 92); the validated engine uses
   **Wilder** RMA (`scalper_backtest.py::wilder_atr`). Everything is ATR-scaled, so
   the live trade set partially diverges from the validated set (74–83% signal overlap
   measured per symbol on real Deriv M15).
2. **Risk-guard state reset on every re-init** — OnInit rebased the 4% daily ledger and
   zeroed the trade counter (a mid-day restart could allow ~2× the daily budget).
3. **Trailing-only drawdown floor** — peak-since-attach, never anchored to the
   account's initial balance; a reload in drawdown sank the kill line below FTMO's 90% line.
4. **Halts never cancelled resting pendings** — a limit could fill after
   "paused for the day" and add fresh risk.
5. **`CalendarValueHistory` bool treated as a count** — the active news guard inspected
   at most `vals[0]` and crashes the EA on true-with-empty-array (currently a silent
   no-op on the FTMO feed; crash armed if the calendar wakes).

## Tier A fix — v1.26 → v1.26.2 (deployed, compiled 0/0; NO trade-set impact)

- News-guard rewritten (bool check + `ArraySize`); crash removed, all events inspected.
- `CancelAllPendings()` on daily-loss/max-DD/static-floor halts + halted guard in
  `ManagePendingOrders`; max-DD halt is now HARD (survives day rollover).
- `RestoreRiskLedger()` on init: reconstructs today's PnL/placements from deal history,
  persists peak equity via terminal global, adds a **static floor**
  (`InpInitialBalance=100000`, `InpStaticFloorPct=9`) under the trailing check.
  v1.26.1: one-shot re-sync ~60 s after cold start (deal history syncs late — observed
  $495 short at start+5 s). v1.26.2: placements count entry-type orders only.
- `OnTick` gated off under timer+bar-close config (was running full management incl.
  a full-day `HistorySelect` on every tick of the 24/7 host chart).
- Trade-log integrity: reload/prune-race rows now rebuilt from position/deal history
  (was `dir=SELL, entry=0, R=0`); confirmed-delete-only sigATR prune; fallback-ATR
  overwrite when the real frozen ATR arrives; placement gated on `ResultRetcode`.
- `SYMBOL_TRADE_TICK_VALUE_LOSS` in sizing; min-lot over-risk WARN; iATR retry throttle;
  dead `curTP` removed; stale Deriv comments cleaned.

## ATR-style study (`backtest/atr_style_study.py`)

Same audited engine, only the ATR function swapped, 12-symbol spread-gated Deriv M15,
deployed params, 70/30 IS/OOS:

| pool | Wilder OOS | SMA (=iATR) OOS |
|---|---|---|
| 12-symbol pooled | **+0.0518 R** (n=10,167, t=+3.53) | +0.0401 R (n=11,033, t=+2.83) |
| US30+US100 (FTMO pair) | **+0.0753 R** (n=1,617, t=+1.99) | +0.0501 R (n=1,762, t=+1.40) |

**Verdict:** the deployed SMA variant retains a real OOS edge (not broken), but Wilder
is consistently better — +50% relative expectancy on the FTMO pair. Recommended as the
headline Tier B change (self-computed Wilder ATR in `ReadAtr`), bundled with:
bar-counted pending expiry, fills-not-placements day counter, wrong-side-limit
handling (observed live: 10015 reject on a gapped retrace), exit-bar re-signal skip.
Tier B changes the live trade set → deliberate re-baseline of the forward test.
